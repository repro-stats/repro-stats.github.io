Qualifying regulog in a Validated R Environment

Most R packages require weeks of internal validation work before a QA team will sign off on them for use in a regulated environment. regulog ships pre-written, executable IQ/OQ/PQ qualification protocols. This post explains what they cover, how to run them, and what to do with the output.

Why qualification is required

Any software used in a regulated environment — one that falls under 21 CFR Part 11, EU Annex 11, or GAMP 5 — must be formally qualified before it can be used to generate, modify, or sign electronic records that regulators may inspect. This applies to R packages used in clinical data analysis, regulatory submissions, or GxP quality systems just as much as it applies to validated LIMS or EDC systems.

Qualification follows a three-phase structure that has been standard in the pharmaceutical industry for decades:

  • Installation Qualification (IQ) — evidence that the software is installed correctly in this specific environment, with the correct version and its dependencies intact.
  • Operational Qualification (OQ) — evidence that the software operates as specified across all relevant functions, including boundary conditions, negative tests, and every control required by the applicable regulation.
  • Performance Qualification (PQ) — evidence that the software performs reliably under realistic, end-to-end conditions representative of actual use in this environment.

In practice, the bottleneck is almost always the protocol authoring step. Writing IQ/OQ/PQ scripts from scratch, getting them reviewed and approved, and assembling the traceability matrix that links each test to a regulatory requirement is time-consuming work — even for straightforward packages.

What regulog ships

regulog includes pre-written, executable qualification scripts for all three phases. They are version-controlled alongside the package, updated with every release, and produce a self-contained pass/fail record your team can retain as documented evidence of system qualification.

What the protocols cover

The OQ is the most substantive protocol. Each of its 26 tests maps explicitly to a regulatory clause and includes an acceptance criterion. The traceability from test to requirement is documented in the Requirements Traceability Matrix (RTM) included with the package: 

# View the RTM
read.csv(system.file("validation/RTM_regulog.csv", package = "regulog"))

Running the qualification

Run all three protocols in sequence in the target environment — the machine and R installation that will be used for regulated work. Each script is self-contained and requires only that regulog is installed. 

# Phase 1: Installation Qualification
source(system.file("validation/IQ_regulog.R", package = "regulog"))

# Phase 2: Operational Qualification
source(system.file("validation/OQ_regulog.R", package = "regulog"))

# Phase 3: Performance Qualification
source(system.file("validation/PQ_regulog.R", package = "regulog"))

Capturing the qualification record

The output of each protocol — including the platform, R version, date, and pass/fail status of every test — should be retained as your documented evidence of system qualification. The simplest approach is to capture it directly: 

# Capture and save the full qualification output
sink("IQ_execution_record.txt")
source(system.file("validation/IQ_regulog.R", package = "regulog"))
sink()

sink("OQ_execution_record.txt")
source(system.file("validation/OQ_regulog.R", package = "regulog"))
sink()

sink("PQ_execution_record.txt")
source(system.file("validation/PQ_regulog.R", package = "regulog"))
sink()

Each execution record includes the timestamp, R version, platform, and full test results. These files, along with the protocol scripts themselves and the RTM, form the core of a qualification dossier that can be presented during an inspection or internal audit.

Re-qualification on upgrade

Any significant change to the software — a new package version, a change to the R environment, or a platform migration — triggers re-qualification. Re-running the three protocols in the updated environment and retaining the new execution records is sufficient evidence that the qualified state has been re-established.

What the OQ actually tests

The OQ is the most directly relevant protocol for regulatory review. Its 26 tests are grouped by the clause they address:

§11.10(e) — Audit trail requirements

  • Genesis hash is correctly formed and 64 hex characters
  • Each entry links cryptographically to its predecessor
  • Content modification in any entry is detected
  • Entry insertion, deletion, or reordering is detected
  • User identity is preserved on every entry
  • Timestamps are ISO-8601 UTC format
  • Blank reason fields are rejected — no silent entries possible
  • log_note() creates hash-chained NOTE entries
  • with_log() isolates errors without corrupting the chain

§11.10(b) — Record retrieval and export

  • CSV export contains all required regulatory columns
  • Signed export includes chain_intact and verified_at fields
  • Date filtering correctly excludes out-of-range entries
  • filter_log() returns only entries of the requested type
  • Genesis record is excluded from user-facing exports

§11.10(c) — Record protection

  • Entries persist to disk and verify correctly from the .rlog file path without an active session

§11.100 / §11.200 — Electronic signatures

  • Signer identity and meaning are recorded on every signature entry
  • Entries covered by the signature are counted automatically
  • Signature entries are part of the hash chain
  • Tampering with any entry before the signature is detected

Using regulog to audit the qualification itself

There is a useful circularity here: the qualification run of regulog is itself an activity in a regulated environment, and it should therefore be logged. You can use regulog to create an audit trail of the qualification process: 

library(regulog)

# Open a session for the qualification run
log <- regulog_init(
  app     = "regulog-qualification",
  version = "0.2.0",
  user    = "val.lead",
  path    = "qualification/audit_trail.rlog"
)

log_action(log,
  action = "qualification_start",
  object = "regulog 0.2.0",
  reason = "IQ/OQ/PQ qualification initiated per SOP-VAL-007"
)

# Run IQ
source(system.file("validation/IQ_regulog.R", package = "regulog"))
log_action(log,
  action = "IQ_complete",
  object = "IQ_regulog.R",
  reason = "10 tests passed. Proceeding to OQ."
)

# Run OQ
source(system.file("validation/OQ_regulog.R", package = "regulog"))
log_action(log,
  action = "OQ_complete",
  object = "OQ_regulog.R",
  reason = "26 tests passed. Proceeding to PQ."
)

# Run PQ
source(system.file("validation/PQ_regulog.R", package = "regulog"))
log_action(log,
  action = "PQ_complete",
  object = "PQ_regulog.R",
  reason = "7 tests passed. Qualification complete."
)

# Sign off
log_signature(log,
  "I certify that regulog 0.2.0 has been qualified in this environment
   per SOP-VAL-007 and is approved for use in regulated R workflows."
)

# Verify and export the qualification audit trail
verify_log(log)
export_audit_trail(log,
  format = "csv",
  signed = TRUE,
  path   = "qualification/audit_trail_export.csv"
)

The resulting audit_trail_export.csv is itself a Part 11-compliant electronic record documenting the qualification activity — who ran it, when, and that the outcome was reviewed and signed.

Bottom line

The IQ/OQ/PQ protocols shipped with regulog reduce the qualification burden from weeks of protocol authoring to three commands. The output is auditable, traceable to regulatory requirements, and produced in the environment where the package will actually be used — which is exactly what an inspector expects to see.
